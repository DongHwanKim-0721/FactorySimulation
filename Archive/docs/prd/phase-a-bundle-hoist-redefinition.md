# PRD: 단계 A 재정의 - 묶음/호이스트 기반 공정 시뮬레이션

## Problem Statement

기존 단계 A 구현은 EA가 각 공정에서 하나씩 완료되는 즉시 다음 공정으로 흘러가는 push 모델로 정의됐다. 이 모델은 `capacity`를 병렬 EA 슬롯으로 해석하고, 다음 공정은 앞 공정의 개별 EA 완료 시각에 맞춰 곧바로 시작한다. 테스트 관점에서는 일관적이지만, 강관 인발 현장의 실제 흐름과 맞지 않는다.

강관 인발 공정에서는 소재가 공정 사이를 EA 단위로 즉시 이동하지 않는다. 전처리 같은 앞 공정이 대상 물량을 모두 처리한 뒤, 호이스트 크레인이 소재를 몇 차례 왕복해 뭉텅이로 다음 설비 앞에 옮긴다. 인발기 같은 공정 설비는 원자재 품종에 따라 금형이 달라지므로, 같은 설비에 여러 품종이 합류하더라도 A 품종을 처리하다가 B 품종을 끼워 넣고 다시 A 품종으로 돌아오는 방식은 바람직하지 않다.

또한 `시작 공정별 투입 수량(EA)`라는 전역 입력은 실제 원자재 흐름을 표현하기에 애매하다. 현장에서는 A 원자재와 B 원자재가 각각 다른 라인에서 투입되고 전처리된 뒤 같은 인발기로 합류할 수 있다. 따라서 원자재 수량은 전역값이 아니라 원자재 투입 블록이 소유해야 한다.

결과적으로 단계 A는 "EA 개별 push 스케줄러"가 아니라 "원자재 투입 블록이 생성한 묶음이 공정/호이스트 블록을 거치며 이동하는 묶음 기반 스케줄러"로 재정의되어야 한다.

## Solution

단계 A를 묶음 기반 deterministic 스케줄러로 재정의한다. 시뮬레이션 단위는 개별 EA가 아니라 `묶음(Bundle)`이다. 묶음은 원자재 투입 블록에서 생성되고, 원자재명과 EA 수량을 끝까지 들고 다닌다.

사용자는 기존 `원자재 투입` 블록을 그대로 사용하되, 이 블록의 의미를 일반 공정에서 소스 블록으로 바꾼다. 원자재 투입 블록은 `원자재명`, `투입 원자재 수(EA)`, `투입 시간(분)`을 가진다. 원자재 투입 블록은 입력 간선 없는 시작 블록으로만 허용하며 여러 개 존재할 수 있다.

일반 공정 블록은 `처리 시간(분/EA)`과 `동시 가공 수량(EA)`을 가진다. 공정 블록은 받은 묶음별로 소요 시간을 계산한다. 한 묶음의 공정 소요 시간은 `ceil(묶음 EA 수 / 동시 가공 수량) * 처리 시간`이다.

호이스트 블록을 명시적인 블록 타입으로 추가한다. 호이스트 블록은 `1회 운반 수량(EA)`과 `1회 이동 시간(분)`을 가진다. 호이스트 소요 시간은 `ceil(묶음 EA 수 / 1회 운반 수량) * 1회 이동 시간`이다.

직렬 흐름에서는 앞 블록이 묶음 전체를 완료한 뒤 다음 블록이 시작한다. 호이스트도 동일하게 묶음 전체 이동이 완료된 뒤 다음 블록으로 넘긴다. 즉, 기존처럼 EA가 하나씩 다음 공정으로 즉시 push되지 않는다.

분기는 이번 단계에서는 자동 분배 규칙을 유지한다. 후속 블록의 능력치를 가중치로 사용해 라우팅 패턴을 만든다. 후속 블록이 공정이면 `동시 가공 수량(EA)`, 호이스트면 `1회 운반 수량(EA)`를 가중치로 사용한다. 예를 들어 후속 B의 가중치가 4, C의 가중치가 1이면 `B, B, B, B, C` 패턴을 반복한다. 이 규칙은 C 단계에서 연결별 직접 수량/비율 지정으로 확장할 수 있다.

합류는 묶음 단위 FIFO를 따른다. 여러 라인에서 같은 공정으로 묶음이 도착하면 먼저 도착한 묶음 전체가 먼저 처리된다. 묶음은 병합하지 않는다.

단, 일반 공정 블록에는 원자재 품종 그룹 우선 처리 규칙을 적용한다. 같은 설비에 여러 품종이 도착하면 가장 먼저 처리하기 시작한 원자재명을 현재 작업 품종으로 보고, 그 품종의 모든 묶음을 먼저 처리한다. 나중에 같은 품종 묶음이 추가로 도착할 예정이면 설비가 대기하더라도 해당 품종을 마저 처리한 뒤 다음 품종으로 넘어간다. 이 규칙은 금형 전환을 줄이기 위한 기본 정책이다. 호이스트 블록에는 이 품종 그룹 우선 규칙을 적용하지 않고 도착 묶음 FIFO만 적용한다.

전체 투입 수량은 모든 원자재 투입 블록의 투입 원자재 수 합계다. 최종 output 수량은 출력 간선이 없는 마지막 블록들의 완료 수량 합계다. 불량, 폐기, 조립, BOM, 수율이 없는 한 최종 output 수량은 전체 투입 수량과 같아야 한다.

## User Stories

1. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 수량을 전역 입력이 아니라 원자재 투입 블록에서 입력하기를 원한다, so that 여러 종류의 원자재가 동시에 또는 독립 라인으로 들어오는 현장을 표현할 수 있다.
2. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 블록에 원자재명을 입력하기를 원한다, so that 이후 공정에서 품종별 작업 순서를 구분할 수 있다.
3. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 블록에 투입 원자재 수(EA)를 입력하기를 원한다, so that 전체 투입량과 최종 output 수량을 명확히 추적할 수 있다.
4. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 블록에 투입 시간(분)을 입력하기를 원한다, so that 원자재 묶음을 준비하거나 투입하는 시간을 시뮬레이션에 반영할 수 있다.
5. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 시간이 전체 묶음에 걸리는 고정 시간으로 계산되기를 원한다, so that 원자재 투입을 EA별 공정이 아니라 준비 작업으로 모델링할 수 있다.
6. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 블록을 여러 개 배치하기를 원한다, so that A 원자재 라인과 B 원자재 라인을 각각 모델링할 수 있다.
7. As a 강관 인발 시뮬레이션 사용자, I want 원자재 투입 블록에는 입력 연결을 만들 수 없기를 원한다, so that 원자재 투입이 항상 수량을 생성하는 시작점임을 보장할 수 있다.
8. As a 강관 인발 시뮬레이션 사용자, I want 일반 공정 블록은 입력 연결 없이 시작할 수 없기를 원한다, so that 모든 공정 수량이 원자재 투입 블록에서 비롯됨을 보장할 수 있다.
9. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 블록은 입력 연결 없이 시작할 수 없기를 원한다, so that 이동 대상 없는 호이스트 작업이 계산되지 않는다.
10. As a 강관 인발 시뮬레이션 사용자, I want 공정 블록의 `capacity`를 `동시 가공 수량(EA)`으로 이해하기를 원한다, so that 설비가 한 번에 몇 EA를 가공할 수 있는지 명확히 표현할 수 있다.
11. As a 강관 인발 시뮬레이션 사용자, I want 공정 블록의 처리 시간이 `처리 시간(분/EA)`로 유지되기를 원한다, so that 현장에서 쓰는 EA당 처리 시간 기준을 그대로 사용할 수 있다.
12. As a 강관 인발 시뮬레이션 사용자, I want 공정 블록의 묶음 소요 시간이 `ceil(묶음 EA 수 / 동시 가공 수량) * 처리 시간`으로 계산되기를 원한다, so that 10EA 묶음이 동시 가공 수량에 맞춰 누적 처리되는 시간을 볼 수 있다.
13. As a 강관 인발 시뮬레이션 사용자, I want 앞 공정 묶음 전체가 완료된 뒤 다음 블록이 시작하기를 원한다, so that 소재가 공정 사이에서 하나씩 흘러가지 않고 묶음으로 이동하는 현장을 반영할 수 있다.
14. As a 강관 인발 시뮬레이션 사용자, I want `공정1 -> 공정2`에서 10EA, 각 공정 1분/EA, 동시 가공 수량 1EA이면 공정 시간이 10분 + 10분으로 계산되기를 원한다, so that 누적 처리 모델이 기대대로 동작함을 확인할 수 있다.
15. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 크레인을 명시적 블록으로 배치하기를 원한다, so that 공정 사이 소재 이동 시간이 결과에 드러난다.
16. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 블록에 1회 운반 수량(EA)을 입력하기를 원한다, so that 소재를 몇 EA씩 묶어 옮기는지 반영할 수 있다.
17. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 블록에 1회 이동 시간(분)을 입력하기를 원한다, so that 호이스트 왕복 또는 이동에 걸리는 시간을 반영할 수 있다.
18. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 소요 시간이 `ceil(묶음 EA 수 / 1회 운반 수량) * 1회 이동 시간`으로 계산되기를 원한다, so that 2~3번 왕복해 소재 이동이 끝나는 흐름을 표현할 수 있다.
19. As a 강관 인발 시뮬레이션 사용자, I want 호이스트가 묶음 전체 이동을 완료한 뒤 다음 공정이 시작하기를 원한다, so that 인발기 투입 전에 소재 이동이 완료되는 현장 흐름을 반영할 수 있다.
20. As a 강관 인발 시뮬레이션 사용자, I want 분기에서 EA가 복제되지 않기를 원한다, so that 최종 output 수량이 실제 투입 수량보다 커지는 오류를 막을 수 있다.
21. As a 강관 인발 시뮬레이션 사용자, I want 분기 수량이 후속 블록의 능력치 가중치로 자동 분배되기를 원한다, so that 직접 분기 비율 UI가 없는 상태에서도 예측 가능한 분배 결과를 얻을 수 있다.
22. As a 강관 인발 시뮬레이션 사용자, I want 후속 공정 블록의 분기 가중치가 동시 가공 수량(EA)이기를 원한다, so that 더 많은 동시 가공 능력을 가진 라인에 더 많은 물량이 배정된다.
23. As a 강관 인발 시뮬레이션 사용자, I want 후속 호이스트 블록의 분기 가중치가 1회 운반 수량(EA)이기를 원한다, so that 더 많이 운반할 수 있는 호이스트 경로에 더 많은 물량이 배정된다.
24. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 블록 다음에도 분기가 가능하기를 원한다, so that 한 호이스트 이동 후 여러 설비로 나눠 보내는 흐름을 표현할 수 있다.
25. As a 강관 인발 시뮬레이션 사용자, I want 합류 공정에서 묶음별 FIFO가 유지되기를 원한다, so that 먼저 도착한 묶음 전체가 먼저 처리된다.
26. As a 강관 인발 시뮬레이션 사용자, I want 합류 공정에서 서로 다른 묶음이 자동 병합되지 않기를 원한다, so that 어떤 원자재 라인에서 온 묶음인지 추적할 수 있다.
27. As a 강관 인발 시뮬레이션 사용자, I want 공정 블록에서 같은 원자재 품종을 몰아서 처리하기를 원한다, so that 금형 교체가 잦아지는 비현실적 순서를 피할 수 있다.
28. As a 강관 인발 시뮬레이션 사용자, I want A 원자재 묶음을 처리하기 시작했다면 나중에 도착할 A 원자재 묶음까지 기다려 A를 끝낸 뒤 B 원자재를 처리하기를 원한다, so that 같은 인발기에서 원자재별 금형 흐름을 유지할 수 있다.
29. As a 강관 인발 시뮬레이션 사용자, I want 품종 그룹 우선 처리가 모든 공정 블록에 기본 적용되기를 원한다, so that 별도 옵션 설정 없이도 품종 섞임을 기본적으로 방지할 수 있다.
30. As a 강관 인발 시뮬레이션 사용자, I want 호이스트 블록에는 품종 그룹 우선 처리가 적용되지 않기를 원한다, so that 운반 장비는 먼저 도착한 묶음 기준으로 단순하게 운반된다.
31. As a 강관 인발 시뮬레이션 사용자, I want 전체 투입 수량이 모든 원자재 투입 블록 수량의 합계로 표시되기를 원한다, so that 여러 원자재 라인이 있어도 전체 투입량을 바로 확인할 수 있다.
32. As a 강관 인발 시뮬레이션 사용자, I want 최종 output 수량이 출력 간선이 없는 마지막 블록들의 완료 수량 합계로 표시되기를 원한다, so that 별도 제품 출력 블록 없이도 산출량을 확인할 수 있다.
33. As a 강관 인발 시뮬레이션 사용자, I want 불량이나 폐기가 없는 한 최종 output 수량이 전체 투입 수량과 같기를 원한다, so that 기본 시뮬레이션에서 수량 보존이 보장된다.
34. As a 강관 인발 시뮬레이션 사용자, I want 결과에서 블록별 실제 처리 수량(EA)을 확인하기를 원한다, so that 각 공정과 호이스트를 통과한 물량을 검증할 수 있다.
35. As a 강관 인발 시뮬레이션 사용자, I want 결과에서 블록별 처리 묶음 수를 확인하기를 원한다, so that 같은 블록이 몇 개 묶음을 처리했는지 이해할 수 있다.
36. As a 강관 인발 시뮬레이션 사용자, I want 결과에서 묶음별 원자재명과 수량을 확인하기를 원한다, so that 합류 이후에도 어떤 품종이 어떤 순서로 처리됐는지 추적할 수 있다.
37. As a 강관 인발 시뮬레이션 사용자, I want 결과에서 호이스트 이동 횟수를 확인하기를 원한다, so that 1회 운반 수량이 전체 리드타임에 미치는 영향을 이해할 수 있다.
38. As a 강관 인발 시뮬레이션 사용자, I want 잘못된 블록 연결을 즉시 알림받기를 원한다, so that INPUT으로 들어가는 연결이나 시작점 없는 공정을 만들지 않는다.
39. As a 개발자, I want 시뮬레이션 엔진이 GUI 없이 호출 가능한 순수 모듈로 유지되기를 원한다, so that 묶음 스케줄링 규칙을 자동 테스트로 검증할 수 있다.
40. As a 개발자, I want 묶음 스케줄러가 작은 공개 인터페이스 뒤에 숨겨지기를 원한다, so that 내부 큐와 라우팅 구현이 바뀌어도 GUI 호출 방식은 안정적으로 유지된다.
41. As a 개발자, I want `capacity` 같은 모호한 필드를 도메인별 명확한 필드로 바꾸기를 원한다, so that 공정의 동시 가공 수량과 호이스트의 운반 수량이 혼동되지 않는다.
42. As a 개발자, I want 저장 포맷을 새 의미 모델에 맞춰 단순하게 바꾸기를 원한다, so that 구 포맷 호환성 때문에 새 도메인 모델이 꼬이지 않는다.
43. As a 개발자, I want 기존 Phase A의 EA 개별 push 테스트를 새 묶음 기반 골든 테스트로 교체하기를 원한다, so that 잘못 정의된 요구사항을 회귀 기준으로 남기지 않는다.
44. As a 향후 기능 개발자, I want 분기 수량 직접 지정, 제품 출력 블록, BOM, 수율 같은 확장을 C 단계로 분리하기를 원한다, so that 이번 단계의 핵심 스케줄링 모델을 먼저 안정화할 수 있다.

## Implementation Decisions

- 단계 A는 기존 EA 개별 push 스케줄러를 폐기하고 묶음 기반 deterministic 스케줄러로 재구현한다.
- 시뮬레이션의 기본 이동 단위는 `묶음(Bundle)`이다. 묶음은 원자재명, EA 수량, 도착 시각, 원천 INPUT 정보를 가진다.
- 원자재 투입 블록은 기존 블록 타입을 유지하되 일반 공정이 아니라 소스 블록으로 의미를 바꾼다.
- 원자재 투입 블록의 필드는 `원자재명`, `투입 원자재 수(EA)`, `투입 시간(분)`이다.
- 원자재 투입 블록의 소요 시간은 전체 투입 묶음에 걸리는 고정 시간이다. EA당 시간이 아니다.
- 원자재 투입 블록은 여러 개 허용한다.
- 원자재 투입 블록에는 입력 간선을 허용하지 않는다.
- 일반 공정 블록과 호이스트 블록은 입력 간선 없는 시작 블록이 될 수 없다.
- 일반 공정 블록의 필드는 `처리 시간(분/EA)`, `동시 가공 수량(EA)`이다.
- 일반 공정 묶음 소요 시간은 `ceil(묶음 EA 수 / 동시 가공 수량) * 처리 시간`이다.
- 호이스트 블록을 새 명시적 블록 타입으로 추가한다.
- 호이스트 블록의 필드는 `1회 운반 수량(EA)`, `1회 이동 시간(분)`이다.
- 호이스트 묶음 소요 시간은 `ceil(묶음 EA 수 / 1회 운반 수량) * 1회 이동 시간`이다.
- 직렬 흐름에서는 앞 블록이 묶음 전체를 완료한 뒤 다음 블록으로 전달한다.
- 분기에서는 후속 블록의 능력치를 가중치로 사용해 묶음을 쪼갠다.
- 후속 공정 블록의 분기 가중치는 `동시 가공 수량(EA)`이다.
- 후속 호이스트 블록의 분기 가중치는 `1회 운반 수량(EA)`이다.
- 후속 INPUT 블록으로의 연결은 허용하지 않는다.
- 분기 라우팅은 연결 생성 순서와 후속 가중치로 만든 반복 패턴을 사용한다.
- 분기에서 EA는 복제되지 않는다. 분기 결과 묶음들의 EA 수 합은 원 묶음 EA 수와 같아야 한다.
- 호이스트 블록 다음에도 분기를 허용한다.
- 합류는 묶음 단위 FIFO를 따른다. 먼저 도착한 묶음 전체가 먼저 처리된다.
- 묶음은 합류 지점에서 자동 병합하지 않는다.
- 일반 공정 블록은 품종 그룹 우선 처리 규칙을 적용한다.
- 품종 그룹 우선 처리에서는 가장 먼저 처리하기 시작한 원자재명을 현재 작업 품종으로 잡고, 그 품종의 모든 묶음을 처리한 뒤 다음 품종으로 넘어간다.
- 품종 그룹 우선 처리 중 같은 품종의 추가 묶음이 나중에 도착할 예정이면, 공정 블록은 필요하면 대기한 뒤 해당 품종을 계속 처리한다.
- 호이스트 블록에는 품종 그룹 우선 처리 규칙을 적용하지 않는다. 호이스트는 도착 묶음 FIFO만 따른다.
- 전체 투입 수량은 모든 원자재 투입 블록의 투입 원자재 수 합계다.
- 최종 output 수량은 출력 간선이 없는 마지막 블록들의 완료 수량 합계다.
- 불량, 폐기, 조립, BOM, 수율이 없으므로 정상 시뮬레이션에서는 전체 투입 수량과 최종 output 수량이 같아야 한다.
- 결과 모델은 전체 투입 수량, 최종 output 수량, 블록별 처리 EA 수, 블록별 처리 묶음 수, 호이스트 이동 횟수, 묶음별 원자재명/수량/시작/완료 시각을 제공한다.
- 기존 `total_batches`, `units_per_source`, `source_count`, `total_generated_units` 중심 계약은 새 묶음 모델에 맞게 제거하거나 이름을 바꾼다.
- 기존 `capacity` 필드는 내부 모델에서 더 명확한 필드로 바꾼다. 공정의 동시 가공 수량과 호이스트의 1회 운반 수량을 같은 필드로 공유하지 않는다.
- 시나리오 저장 포맷은 새 모델 기준으로 변경한다. 기존 JSON과의 호환성은 보장하지 않는다.
- GUI에서 전역 `시작 공정별 투입 수량(EA)` 입력은 제거한다.
- GUI에서 INPUT 블록 설정 화면은 원자재명, 투입 원자재 수, 투입 시간을 편집하게 바꾼다.
- GUI에서 공정 블록 설정 화면은 처리 시간(분/EA), 동시 가공 수량(EA)을 편집하게 바꾼다.
- GUI에서 호이스트 블록 설정 화면은 1회 운반 수량(EA), 1회 이동 시간(분)을 편집하게 만든다.
- GUI 결과 요약은 전체 투입 수량, 최종 output 수량, 총 소요 시간, 블록별 처리 수량, 묶음 수, 호이스트 이동 횟수를 보여준다.
- 현재의 분기 자동 분배는 임시 규칙이다. 연결별 수량/비율 직접 지정은 C 단계의 도메인 확장으로 남긴다.

## Testing Decisions

- 좋은 테스트는 엔진의 외부 동작만 검증한다. 내부 이벤트 큐, helper 함수 호출 순서, 임시 자료구조에는 의존하지 않는다.
- 묶음 스케줄러는 GUI 없이 직접 호출 가능한 순수 모듈로 테스트한다.
- 데이터 모델 테스트는 블록 타입별 필드 검증, 잘못된 연결 거부, INPUT 입력 간선 거부, 일반/호이스트 시작점 거부를 포함한다.
- 시나리오 직렬화 테스트는 새 JSON 포맷으로 저장/불러오기 후 동일한 시뮬레이션 결과가 나오는지 검증한다.
- GUI 테스트는 자동화하지 않는다. 다만 수동 스모크 체크리스트에는 INPUT 설정, HOIST 설정, 공정 설정, 시뮬레이션 실행, 결과 표시를 포함한다.
- 기존 단계 A 테스트 중 EA 개별 push 모델을 전제로 한 테스트는 새 의미 모델에 맞게 폐기하거나 재작성한다.
- 직렬 누적 골든 테스트를 추가한다. 예: INPUT 10EA, 투입 시간 0분, 공정1 1분/EA 동시 가공 1EA, 공정2 1분/EA 동시 가공 1EA이면 총 공정 시간은 20분이다.
- INPUT 투입 시간 골든 테스트를 추가한다. 예: INPUT 투입 시간 5분이면 동일 직렬 흐름의 총 시간은 25분이다.
- 호이스트 골든 테스트를 추가한다. 예: 10EA, 1회 운반 4EA, 1회 이동 3분이면 호이스트 소요 시간은 9분이고 이동 횟수는 3회다.
- 여러 INPUT 수량 보존 테스트를 추가한다. 예: A INPUT 10EA, B INPUT 5EA이면 전체 투입 수량과 최종 output 수량은 15EA다.
- INPUT은 입력 간선을 가질 수 없다는 테스트를 추가한다.
- 일반 공정과 호이스트가 입력 간선 없이 시작할 수 없다는 테스트를 추가한다.
- 분기 가중치 테스트를 추가한다. 예: 후속 공정 가중치 4, 호이스트 가중치 1이면 10EA가 8EA/2EA로 분배된다.
- 호이스트 이후 분기 테스트를 추가한다. 호이스트 출력 묶음도 동일한 후속 가중치 규칙으로 쪼개져야 한다.
- 합류 묶음 FIFO 테스트를 추가한다. 먼저 도착한 묶음 전체가 먼저 처리되고 나중 묶음은 뒤에 처리되어야 한다.
- 묶음 병합 금지 테스트를 추가한다. 같은 블록에 여러 묶음이 들어와도 결과에서 묶음 수와 원자재명이 유지되어야 한다.
- 공정 품종 그룹 우선 처리 테스트를 추가한다. 예: A 5EA 도착, B 5EA 도착, A 5EA가 나중에 도착하면 공정은 A를 마저 처리한 뒤 B로 넘어가야 한다.
- 호이스트에는 품종 그룹 우선 처리가 적용되지 않는 테스트를 추가한다. 호이스트는 도착 묶음 FIFO로 운반해야 한다.
- 최종 output 수량 테스트를 추가한다. 출력 간선이 없는 마지막 블록들의 완료 EA 합이 전체 투입 수량과 같아야 한다.
- 잘못된 입력 검증 테스트를 추가한다. 투입 원자재 수, 투입 시간, 동시 가공 수량, 1회 운반 수량, 이동 시간은 유효 범위를 벗어나면 명확한 오류를 내야 한다.

## Out of Scope

- 연결별 분기 수량 또는 분기 비율 직접 입력.
- 제품 출력 블록 또는 명시적 OUTPUT 블록.
- 제품 종류, BOM, 조립 규칙.
- A 원자재 1EA + B 원자재 1EA가 제품 1EA가 되는 조립/소모 모델.
- 불량률, 폐기, 수율, 재작업 루프.
- 금형 교체 시간 또는 셋업 시간의 명시적 계산.
- 공정 블록별 품종 그룹 우선 처리 옵션. 이번 단계에서는 모든 공정 블록에 기본 적용한다.
- 호이스트 자원 수량, 호이스트 충돌, 크레인 대기열, 실제 왕복 경로 모델.
- 작업자 자원, 설비 자원 캘린더, 교대 시간.
- 실제 간트 차트, 애니메이션, 결과 비교 모드.
- 기존 JSON 파일과의 호환성.
- 패키징, 설치 프로그램, 실행 파일 빌드.

## Further Notes

- 이 PRD는 기존 단계 A 구현을 보완하는 수준이 아니라 의미 모델을 다시 정의하는 문서다. 기존 단계 A의 EA 개별 push 스케줄러는 강관 인발 현장 모델과 맞지 않는 것으로 판단한다.
- D-A-C-B 순서 자체는 유지하되, A는 "묶음/호이스트 기반 공정 시뮬레이션"으로 재작업해야 한다.
- 이번 단계에서는 분기 자동 분배를 후속 블록 능력치 기반으로 유지한다. 사용자가 직접 수량 또는 비율을 지정하는 기능은 C 단계에서 다루는 것이 적절하다.
- 이번 단계에서는 금형 교체 시간을 계산하지 않는다. 다만 품종 그룹 우선 처리로 금형이 자주 바뀌는 비현실적 순서를 방지한다.
- 현재 레포에는 단계 A 완료 handoff가 존재하지만, 이 PRD가 승인되면 그 handoff는 이전 A 정의의 완료 기록으로만 해석해야 한다.
