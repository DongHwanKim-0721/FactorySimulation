# PRD: 단계 A - 시뮬레이션 엔진 교체

## Problem Statement

단계 D 리팩터링으로 공정 시뮬레이션 로직은 GUI에서 분리됐지만, 현재 엔진은 여전히 레거시 계산 방식을 그대로 따른다. 이 방식에는 사용자가 실제 공정 흐름을 해석할 때 문제가 되는 두 가지 핵심 결함이 있다.

첫째, `capacity`는 화면과 결과에는 표시되지만 실제 스케줄링에서는 동시 처리 수량으로 동작하지 않는다. 예를 들어 동시 처리 수량이 2 EA이고 처리 시간이 10분/EA인 공정에 9 EA가 투입되면 50분에 끝나야 하지만, 현재 방식은 한 번에 하나씩 처리하는 선형 계산에 가깝다.

둘째, 분기와 합류가 있는 그래프를 안정적으로 처리하지 못한다. 사용자가 `A -> B -> D`, `A -> C -> D` 같은 흐름을 그려도, 현재 엔진은 위상정렬 결과를 선형 체인처럼 취급해 실제 흐름과 다른 결과를 낸다.

사용자는 강관 인발 공정의 현실에 맞게 EA 단위의 원자재 수량, 공정별 동시 처리 수량, 호이스트 이동을 고려한 묶음형 분기 흐름을 이해할 수 있어야 한다. 단계 A는 이 목표를 위해 엔진의 의미 모델을 명확히 하고, GUI와 테스트가 그 의미를 일관되게 따르도록 만드는 단계다.

## Solution

시뮬레이션 엔진을 deterministic 직접 이벤트 스케줄러로 교체한다. 외부 이산사건 시뮬레이션 라이브러리는 도입하지 않고, 현재 분리된 엔진 경계 안에서 DAG, FIFO 합류, 병렬 슬롯 처리, capacity 기준 묶음 분기 라우팅을 직접 계산한다.

사용자 관점의 핵심 의미는 다음과 같다.

- 처리 단위는 EA다.
- 처리 시간은 `처리 시간(분/EA)`이다.
- `capacity`는 `동시 처리 수량(EA)`이다.
- 시작 공정에는 사용자가 지정한 `시작 공정별 투입 수량(EA)`이 시간 0에 모두 대기한다.
- 시뮬레이션은 시작 공정에서 생성된 EA가 흐름대로 밀려가는 push 모델이다.
- 중간 공정은 목표 수량을 끌어오지 않고 실제 도착한 EA만 처리한다.
- 분기는 후속 공정의 동시 처리 수량만큼 묶어서 연결 생성 순서대로 보낸다.
- 합류는 여러 부모에서 도착한 EA를 도착 시간순 FIFO 큐로 처리한다.

코드 관점의 핵심 변경은 다음과 같다.

- `total_batches` 명칭을 제거하고 `units_per_source`를 사용한다.
- 결과에는 소스 전체에서 생성된 총 EA 수를 `total_generated_units`로 제공한다.
- 블록별 처리 수량은 기존 공식 계산이 아니라 실제로 해당 블록을 통과한 EA 수로 계산한다.
- 병목은 실제 처리한 블록 중 가장 낮은 이론 처리율을 가진 블록으로 계산한다.
- 그래프 입력은 DAG만 허용하고, cycle과 잘못된 연결은 명확한 입력 오류로 거부한다.

## User Stories

1. As a 공정 시뮬레이션 사용자, I want `capacity`가 동시 처리 수량(EA)으로 실제 스케줄에 반영되기를 원한다, so that 설비가 동시에 처리할 수 있는 EA 수가 총 소요 시간에 정확히 반영된다.
2. As a 공정 시뮬레이션 사용자, I want 처리 시간이 분/EA 단위로 표시되기를 원한다, so that 공정별 입력값이 현장에서 쓰는 수량 단위와 맞게 이해된다.
3. As a 공정 시뮬레이션 사용자, I want 시작 공정별 투입 수량(EA)을 직접 입력하기를 원한다, so that 시뮬레이션 규모를 사용자가 제어할 수 있다.
4. As a 공정 시뮬레이션 사용자, I want 여러 시작 공정이 있을 때 각 시작 공정에 같은 투입 수량이 적용되기를 원한다, so that 복수 원자재 라인을 자연스럽게 표현할 수 있다.
5. As a 공정 시뮬레이션 사용자, I want 전체 투입 수량(EA)이 결과 요약에 표시되기를 원한다, so that 여러 시작 공정이 있을 때 실제 생성된 총 EA 수를 혼동하지 않는다.
6. As a 공정 시뮬레이션 사용자, I want 중간 공정이 실제 도착한 EA만 처리하기를 원한다, so that 분기 후 일부 라인이 적은 수량을 받아도 시뮬레이션이 정상 종료된다.
7. As a 공정 시뮬레이션 사용자, I want 분기에서 EA가 복제되지 않기를 원한다, so that 물리적으로 존재하지 않는 원자재가 결과에 추가되지 않는다.
8. As a 공정 시뮬레이션 사용자, I want 분기된 EA가 후속 공정의 동시 처리 수량만큼 묶여 이동하기를 원한다, so that 호이스트 크레인으로 소재를 묶어서 옮기는 강관 인발 현장의 흐름과 더 가깝게 해석된다.
9. As a 공정 시뮬레이션 사용자, I want 분기 순서가 연결 생성 순서를 따르기를 원한다, so that 내가 먼저 연결한 후속 공정이 먼저 배정되는 예측 가능한 흐름을 얻는다.
10. As a 공정 시뮬레이션 사용자, I want 합류 공정이 여러 부모에서 도착한 EA를 FIFO로 처리하기를 원한다, so that 먼저 도착한 소재가 먼저 처리된다는 직관적인 결과를 얻는다.
11. As a 공정 시뮬레이션 사용자, I want 여러 종료 공정이 있을 때 전체 소요 시간이 가장 늦게 끝난 종료 공정 기준으로 표시되기를 원한다, so that 전체 시나리오 완료 시점을 정확히 알 수 있다.
12. As a 공정 시뮬레이션 사용자, I want 연결 없는 블록도 독립 시작/종료 공정으로 계산되기를 원한다, so that 독립 라인을 간단히 비교할 수 있다.
13. As a 공정 시뮬레이션 사용자, I want 결과에서 블록별 실제 처리 수량(EA)을 확인하기를 원한다, so that 분기 후 각 라인이 얼마나 처리했는지 바로 볼 수 있다.
14. As a 공정 시뮬레이션 사용자, I want 결과에서 이론 처리율이 EA/분 단위로 표시되기를 원한다, so that 공정별 처리 능력을 같은 기준으로 비교할 수 있다.
15. As a 공정 시뮬레이션 사용자, I want 병목 공정이 가장 낮은 이론 처리율 기준으로 표시되기를 원한다, so that 설비 능력 관점의 병목을 빠르게 찾을 수 있다.
16. As a 공정 시뮬레이션 사용자, I want 순환 공정을 만들면 즉시 알림을 받기를 원한다, so that 단계 A에서 지원하지 않는 재작업 루프를 실수로 시뮬레이션하지 않는다.

## Implementation Decisions

- 엔진은 직접 이벤트 스케줄러로 구현한다. 외부 이산사건 시뮬레이션 라이브러리는 단계 A에서 도입하지 않는다.
- 엔진의 공개 호출 개념은 `units_per_source`를 받는 방식으로 바뀐다. 기존 `total_batches` 키워드는 alias 없이 제거한다.
- 결과 모델은 `units_per_source`와 `total_generated_units`를 제공한다. `total_generated_units`는 모든 시작 공정에서 생성된 총 EA 수다.
- 사용자에게 보이는 단위는 EA 중심으로 정리한다. 처리 시간은 `처리 시간(분/EA)`, 동시 처리 수량은 `동시 처리 수량(EA)`, 시작 수량은 `시작 공정별 투입 수량(EA)`로 표시한다.
- `capacity`의 의미는 한 블록에서 동시에 처리 가능한 EA 수다. 구현상 같은 공정 안의 병렬 처리 슬롯으로 해석한다.
- `process_time`의 의미는 EA 1개가 처리 완료되는 데 걸리는 고정 시간이다. 확률 분포나 난수 기반 처리 시간은 지원하지 않는다.
- 소스 공정은 입력 간선이 없는 블록이다. 각 소스 공정 앞에는 시간 0에 `units_per_source` EA가 모두 대기한다.
- 여러 소스 공정은 허용한다. 각 소스가 독립적으로 `units_per_source` EA를 생성하므로 전체 생성 수량은 소스 수와 `units_per_source`의 곱이다.
- 블록이 0개면 빈 결과를 반환한다. `units_per_source=0`도 빈 시뮬레이션으로 처리한다.
- 입력 간선과 출력 간선이 모두 없는 고립 블록은 독립 소스이자 독립 싱크로 처리한다.
- 스케줄링은 push 모델이다. 시작 공정에서만 EA가 생성되고, 하위 공정은 실제 도착한 EA만 처리한다.
- 각 블록은 `capacity`개 처리 슬롯을 가진다. 슬롯 배정은 가장 빨리 비는 슬롯을 우선하고, 동률이면 낮은 슬롯 index를 우선한다.
- `start_times`, `completion_times`, `waiting_times`는 해당 블록에서 처리를 시작한 순서로 기록한다.
- `waiting_times`는 EA가 해당 블록 입력 큐에 도착한 시각부터 처리 시작까지 걸린 시간이다.
- `avg_waiting`은 실제 처리된 EA들의 평균 대기시간이다. 처리 수량이 0이면 0이다.
- `BlockResult.total_processed`는 그 블록을 실제로 통과한 EA 수다. 더 이상 투입 수량과 capacity를 곱하지 않는다.
- `throughput`은 이론 처리율이며 `capacity / process_time`으로 계산한다. UI에는 EA/분 단위로 표시한다.
- 병목은 실제 처리한 블록 중 이론 처리율이 가장 낮은 블록이다. 동률이면 안정 위상 정렬 순서에서 먼저 나온 블록을 선택한다.
- 그래프는 DAG만 허용한다. cycle은 연결 추가 시점과 시뮬레이션 실행 시점 모두에서 입력 오류로 거부한다.
- 존재하지 않는 블록을 가리키는 연결은 시뮬레이션 실행 시점에 입력 오류로 거부한다.
- 같은 부모와 자식 사이의 중복 연결은 시뮬레이션 실행 시점에 입력 오류로 거부한다.
- 안정 위상 정렬을 사용한다. 같은 우선순위의 블록은 기존 블록 목록 순서를 유지한다.
- 분기 라우팅은 후속 공정의 `capacity` 기준 묶음 라우팅이다. 같은 부모에서 나가는 연결은 연결 생성 순서대로 반복한다.
- 분기 예시는 후속 공정 B의 capacity가 4, C의 capacity가 1이면 `B, B, B, B, C` 순서를 반복하는 방식이다.
- 분기 라우팅에서 `process_time`은 비율에 사용하지 않는다. 처리 시간은 스케줄링과 병목 계산에만 사용한다.
- 분기 묶음 라우팅은 실제 호이스트 적재 한계를 모델링하지 않는다. 단계 A에서는 후속 공정의 동시 처리 수량만큼 모아서 보내는 임시 규칙으로 설명한다.
- 합류는 단순 FIFO 큐다. 여러 부모에서 도착한 EA는 도착 시간순으로 처리한다.
- 합류에서 같은 시각에 여러 EA가 도착하면 부모 블록의 안정 위상 정렬 순서, 그다음 해당 부모 안의 완료 순서로 tie-break한다.
- 여러 싱크가 있으면 `total_time`은 모든 싱크의 마지막 완료 시각 중 최대값이다.
- 입력 검증은 `process_time > 0`, `capacity > 0`, `units_per_source >= 0`을 요구한다.
- `process_time=0`인 통과 공정은 단계 A에서 지원하지 않는다.
- GUI는 엔진 호출 전에 시작 공정별 투입 수량을 입력받고, 기본값은 10 EA로 둔다.
- GUI 결과 요약은 시작 공정 수, 시작 공정별 투입 수량(EA), 전체 투입 수량(EA), 총 소요 시간, 병목 공정을 보여준다.
- GUI의 기존 "용량" 표현은 "동시 처리 수량(EA)"로 바꾼다.
- GUI의 처리율 표현은 "이론 처리율: X EA/분"으로 바꾼다.

## Testing Decisions

- 좋은 테스트는 엔진의 외부 동작만 검증한다. 내부 이벤트 큐 구조, 슬롯 자료구조, helper 함수 호출 순서에는 의존하지 않는다.
- 엔진 테스트는 deterministic이어야 한다. 같은 입력은 항상 같은 `process_flow`, 처리 수량, 시간 결과를 반환해야 한다.
- 테스트는 GUI를 띄우지 않고 엔진과 시나리오 모델을 직접 호출한다.
- 기존 단계 D의 선형 골든 테스트는 유지하되 명칭과 필드를 EA 단위로 바꾼다.
- T1 골든: 단일 블록, 처리 시간 30분/EA, 동시 처리 수량 1 EA, 시작 공정별 투입 수량 10 EA이면 총 소요 시간은 300분이고 병목은 해당 블록이다.
- T2 골든: 선형 3블록 흐름에서 처리 시간이 30, 45, 120분/EA이고 각 동시 처리 수량이 1 EA이면 시작 공정별 투입 수량 10 EA 기준 총 소요 시간은 1275분이고 병목은 마지막 블록이다.
- T3 골든: 단일 블록, 처리 시간 30분/EA, 동시 처리 수량 3 EA, 시작 공정별 투입 수량 10 EA이면 총 소요 시간은 120분이다.
- T4 골든: `A -> B -> D`, `A -> C -> D` 구조에서 모든 처리 시간이 10분/EA이고 A=1, B=4, C=1, D=1의 동시 처리 수량을 가진다. 연결은 A에서 B로 먼저, A에서 C로 나중에 생성된다. 시작 공정별 투입 수량 10 EA 기준 라우팅은 `B, B, B, B, C, B, B, B, B, C`가 되고, 처리 수량은 A=10, B=8, C=2, D=10이다.
- T4의 총 소요 시간은 120분으로 고정한다. 이는 push 모델, capacity 기준 묶음 라우팅, FIFO 합류, D의 단일 처리 슬롯을 함께 검증한다.
- cycle 테스트는 시나리오 모델과 엔진 양쪽에 추가한다. 예를 들어 1 -> 2, 2 -> 3 이후 3 -> 1 연결은 거부되어야 한다.
- orphan connection 테스트를 엔진에 추가한다. 존재하지 않는 블록 ID를 가리키는 연결은 입력 오류다.
- duplicate connection 테스트를 엔진에 추가한다. 같은 부모와 자식 사이의 중복 간선은 입력 오류다.
- `units_per_source=0` 테스트를 추가한다. 결과는 총 시간 0, 총 생성 수량 0, 블록별 처리 수량 0, 병목 없음이어야 한다.
- 블록이 0개인 입력 테스트를 추가한다. 결과는 빈 timeline, 총 시간 0, 총 생성 수량 0, 병목 없음이어야 한다.
- 여러 소스 테스트를 추가할 수 있다. 각 소스가 독립적으로 `units_per_source` EA를 생성하고 전체 생성 수량이 소스 수만큼 증가하는지 검증한다.
- GUI는 자동화 테스트 범위 밖이다. 다만 수동 스모크 테스트에서 입력 라벨, 기본값 10 EA, 결과 요약 문구, 기존 블록 편집 문구가 EA 용어로 바뀌었는지 확인한다.

## Out of Scope

- 한 설비가 여러 EA를 하나의 로트로 묶어 처리하는 batch size 또는 lot size 모델은 단계 A 범위 밖이다.
- 소재별 호이스트 적재 한계, 호이스트 이동 시간, 호이스트 대기열, 크레인 자원 수량은 단계 A 범위 밖이다.
- `transfer_batch_size` 필드는 단계 A에서 도입하지 않는다. 나중에 운반 제약을 모델링할 때 `capacity`와 분리한다.
- 확률적 처리 시간, seed, 난수 기반 시뮬레이션은 단계 A 범위 밖이다.
- 재작업 루프, 불량률, 폐기, 재투입 확률, 최대 반복 횟수는 단계 A 범위 밖이다.
- 조립처럼 여러 부모에서 각각 1 EA씩 모여야 다음 공정이 시작되는 동기화 합류는 단계 A 범위 밖이다.
- 분기 비율을 사용자가 직접 입력하는 UI는 단계 A 범위 밖이다.
- 하위 라인의 병목까지 고려한 최적 라우팅 또는 최대 유량 계산은 단계 A 범위 밖이다.
- 실제 간트 차트, 애니메이션, 결과 비교 모드 같은 시각화 강화는 단계 A 범위 밖이다.
- 패키징, 설치 프로그램, 실행 파일 빌드는 단계 A 범위 밖이다.

## Further Notes

- 단계 A는 단계 D에서 확보한 분리 경계를 유지해야 한다. 엔진 교체가 GUI 전체 재작성으로 번지지 않아야 한다.
- 기존 `WholeCode.py.bak`는 단계 A 구현 중 의미 비교용으로 참고할 수 있지만, 단계 A 완료 후에는 새 의미 모델이 레거시 계산과 달라지는 것이 정상이다.
- `capacity`는 코드와 문서에서 동시 처리 수량으로 고정한다. 분기 묶음 라우팅에서도 후속 공정의 동시 처리 수량을 임시 묶음 크기로 쓰지만, 실제 호이스트 적재 한계와는 다른 개념이다.
- 앞선 대화 중 T4 총 소요 시간을 140분으로 언급했지만, 확정된 push 모델과 FIFO 합류 규칙을 적용하면 120분이 맞다. PRD의 골든값은 120분으로 고정한다.
- 단계 A 구현 후 다음 단계에서는 호이스트/운반 제약을 별도 모델로 도입할지, 작업자 자원과 설비 자원을 함께 다룰지 검토할 수 있다.
