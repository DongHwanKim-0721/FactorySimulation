# PRD: 단계 B - UI 개선 및 중앙 다이어그램 애니메이션

Updated: 2026-05-28

Status: ready-for-agent

GitHub references:

- PRD issue: #26

## Problem Statement

현재 FactorySimulation은 단계 D 리팩터링, 단계 A 묶음/호이스트 기반 스케줄러, 단계 C 제품명 라벨 추적을 거치며 계산 모델은 강관 인발 공정의 핵심 흐름을 표현할 수 있게 됐다. 하지만 tkinter GUI는 여전히 정적 다이어그램과 긴 텍스트 결과 중심이다. 사용자는 묶음이 언제 투입되고, 어느 공정에서 대기하며, 어떤 블록이 병목으로 작동하는지 시뮬레이션 결과를 머릿속으로 해석해야 한다.

특히 단계 A 이후 시뮬레이션 단위가 개별 EA가 아니라 제품명/원자재명/수량을 가진 묶음(Bundle)이 됐고, 단계 C 이후 제품 라벨까지 추적된다. 이 정보를 표와 텍스트로만 보여주면 분기, 합류, 품종 그룹 우선 처리, 호이스트 이동 흐름을 직관적으로 파악하기 어렵다.

기존 PRD에는 단계 B가 시각화/UX 강화로 넓게 언급되어 있지만, 이번 단계의 실제 목표는 프레임워크 교체나 차트 라이브러리 도입이 아니다. 사용자는 기존 tkinter 앱을 유지하면서, 운영 도구처럼 조용하고 밀도 있는 UI 개선과 중앙 공정 다이어그램 위의 시뮬레이션 애니메이션을 원한다.

## Solution

기존 tkinter 데스크톱 앱과 `왼쪽 팔레트 / 중앙 공정 다이어그램 / 오른쪽 결과 패널`의 3영역 구조를 유지한다. 화면을 전면 교체하지 않고, 중앙 공정 다이어그램을 시뮬레이션 재생의 주 무대로 개선한다.

시뮬레이션 실행 후 중앙 다이어그램 상단에 playhead 기반 애니메이션 컨트롤을 표시한다. 사용자는 재생, 일시정지, 정지, 속도 선택, 시간 슬라이더를 통해 현재 시뮬레이션 시각을 조작할 수 있다. 기본 1x 재생은 전체 시뮬레이션을 약 30초에 재생하는 적응형 속도로 정의하고, 0.5x/1x/2x/5x는 그 기준에 대한 상대 배율로 동작한다.

애니메이션은 엔진의 기존 `SimulationResult`를 변경하지 않고 GUI에서 해석한다. `BundleRecord` 단위로 토큰을 만들고, 각 토큰은 제품명, 원자재명, EA 수량, 현재 상태를 가진다. 토큰 색상은 제품명 기준으로 안정적으로 배정하고, 원자재명은 보조 텍스트와 상세 정보로 표시한다.

현재 시뮬레이션 시각에 따라 묶음 토큰은 도착 전, 대기 중, 처리 중, 완료 상태로 표시된다. 대기 묶음은 블록 왼쪽 또는 상단에 작게 쌓고, 처리 중 묶음은 블록 안에 표시하며, 완료된 sink 묶음은 완료 시점에 남겨 둔다. 처리 중인 블록은 테두리와 진행률로 강조한다. 병목 블록은 항상 작은 병목 배지를 표시하고, 재생 중 실제 처리 중이면 추가 강조한다.

우측 결과 패널은 완전 재작성하지 않는다. 기존 요약/타임라인/분석 구조를 유지하되, 중앙 애니메이션을 보조하는 현재 시점 정보와 선택 묶음 상세 표시를 추가한다. 긴 텍스트 분석은 보존하되, 사용자가 애니메이션과 함께 확인할 핵심 지표를 더 쉽게 볼 수 있게 정리한다.

## User Stories

1. As a 공정 시뮬레이션 사용자, I want 기존 tkinter 앱을 그대로 실행하기를 원한다, so that 새 UI 기능 때문에 설치 방식이나 실행 방식이 바뀌지 않는다.
2. As a 공정 시뮬레이션 사용자, I want 기존 왼쪽 팔레트, 중앙 다이어그램, 오른쪽 결과 패널 구조가 유지되기를 원한다, so that 익숙한 작업 흐름을 잃지 않는다.
3. As a 공정 시뮬레이션 사용자, I want 중앙 공정 다이어그램 위에서 묶음 이동을 재생하기를 원한다, so that 텍스트 결과를 읽지 않고도 공정 흐름을 이해할 수 있다.
4. As a 공정 시뮬레이션 사용자, I want 시뮬레이션 시간축을 playhead로 조작하기를 원한다, so that 원하는 시점의 공정 상태를 직접 확인할 수 있다.
5. As a 공정 시뮬레이션 사용자, I want 현재 시각과 전체 소요 시간이 함께 표시되기를 원한다, so that 재생 위치가 전체 시뮬레이션 중 어디인지 알 수 있다.
6. As a 공정 시뮬레이션 사용자, I want 재생과 일시정지를 같은 흐름에서 조작하기를 원한다, so that 결과를 보다가 관심 지점에서 멈출 수 있다.
7. As a 공정 시뮬레이션 사용자, I want 정지 버튼으로 애니메이션을 처음 상태로 되돌리기를 원한다, so that 같은 결과를 반복해서 볼 수 있다.
8. As a 공정 시뮬레이션 사용자, I want 시간 슬라이더로 임의 시점으로 이동하기를 원한다, so that 병목이나 대기 발생 시점을 빠르게 찾아볼 수 있다.
9. As a 공정 시뮬레이션 사용자, I want 슬라이더를 직접 움직이면 재생이 일시정지되기를 원한다, so that 사용자가 선택한 시점이 자동 재생으로 지나가지 않는다.
10. As a 공정 시뮬레이션 사용자, I want 1x 재생이 시나리오 총 시간과 관계없이 적당한 길이로 재생되기를 원한다, so that 긴 시나리오도 기다리지 않고 볼 수 있다.

## Implementation Decisions

- 기존 tkinter 데스크톱 앱을 유지한다. 웹 UI, Qt, Electron, 다른 GUI 프레임워크로 전환하지 않는다.
- 기존 3영역 구조를 유지한다. 왼쪽 팔레트는 블록 추가, 중앙 다이어그램은 편집과 애니메이션, 오른쪽 결과 패널은 요약과 상세 분석을 담당한다.
- 중앙 공정 다이어그램을 애니메이션의 1순위 화면으로 둔다. 우측 타임라인 또는 Gantt 스타일 차트는 보조 영역이며 이번 단계의 핵심 구현 대상이 아니다.
- 중앙 다이어그램 상단에 애니메이션 컨트롤 바를 둔다. 전역 툴바에는 기존 실행, 저장, 불러오기, 초기화 흐름을 유지한다.
- 애니메이션 컨트롤은 재생/일시정지, 정지, 속도 선택, 시간 슬라이더, 현재 시각/총 시각 표시를 제공한다.
- 기본 1x 재생은 전체 시뮬레이션을 약 30초에 재생하는 적응형 속도다. 0.5x, 2x, 5x는 이 기준에 대한 상대 배율이다.
- 사용자가 시간 슬라이더를 직접 조작하면 재생은 일시정지되고, 선택된 시각의 토큰 상태가 즉시 렌더링된다.
- 애니메이션은 기존 시뮬레이션 결과를 GUI에서 해석해 만든다. 엔진 결과 모델에는 필드를 추가하지 않는다.
- 토큰의 기본 단위는 엔진 결과의 묶음 레코드다. 제품명, 원자재명, 수량, 블록별 도착/시작/완료 시각을 사용한다.
- 토큰 색상은 제품명 기준으로 안정적으로 배정한다. 원자재명은 토큰 상세, 우측 패널, 보조 텍스트에서 표시한다.
- 한 시각에서 토큰 상태는 도착 전, 대기 중, 처리 중, 완료로 나눈다.
- 대기 중 토큰은 블록 왼쪽 또는 상단에 작게 쌓아 표시한다.
- 처리 중 토큰은 블록 내부에 표시하고, 블록 테두리 또는 진행률로 처리 중임을 보여준다.
- HOIST 블록도 공정 블록과 동일한 상태 표현을 사용한다. 별도의 이동 애니메이션 의미를 부여하지 않는다.
- 연결선 위 이동은 계산 의미가 아니라 짧은 시각 효과로만 다룬다. 실제 이동 시간은 HOIST 블록의 처리 시간으로 표현된다.
- 재생 완료 후에는 전체 완료 시점에 멈춘다. sink 블록 주변에는 완료 토큰 또는 완료 수량을 남긴다.
- 병목 블록은 항상 작은 배지로 표시한다. 현재 시점에 병목 블록이 처리 중이면 추가 강조한다.
- 토큰 클릭은 선택 고정 동작이다. 선택된 토큰은 강조되고, 오른쪽 패널에는 묶음 상세가 표시된다.
- 빈 캔버스 클릭 또는 Escape는 선택 해제로 동작한다.
- 활성 토큰이 일정 개수 이하일 때는 전체 토큰을 표시한다. 활성 토큰이 일정 개수를 초과하면 블록별/상태별 집계 토큰으로 자동 축약한다.
- 축약 모드가 켜지면 상태 영역에 축약 표시 중임을 안내한다.
- 구조 또는 파라미터 변경이 발생하면 애니메이션은 즉시 정지하고 결과를 오래됨 상태로 표시한다.
- 블록 위치 이동은 예외다. 위치 이동은 결과 의미를 바꾸지 않으므로 애니메이션만 정지하고 기존 결과는 유지할 수 있다.
- 블록 추가/삭제, 연결 추가/삭제, 파라미터 변경, 불러오기, 초기화는 재시뮬레이션 필요 상태로 전환한다.
- 저장은 시뮬레이션 결과와 애니메이션 상태에 영향을 주지 않는다.
- 애니메이션 상태는 시나리오 저장 파일에 저장하지 않는다. 현재 재생 시간, 속도, 선택 묶음, 토큰 위치 캐시는 저장 대상이 아니다.
- UI 스타일은 운영 도구에 맞게 조용하고 밀도 있게 개선한다. 밝은 회색 작업 배경, 흰 패널, 절제된 포인트 컬러, 명확한 연결선, 정보 계층이 분명한 블록 표현을 사용한다.
- 캔버스에는 배경 그리드와 더 명확한 선택/연결 모드 피드백을 추가한다.
- 자동 배치, 강제 snap-to-grid, 미니맵, 대규모 줌/팬 재작업은 이번 단계에 포함하지 않는다.
- 결과 패널은 완전 재작성하지 않는다. 기존 요약/타임라인/분석 구조를 유지하되, 현재 시점 정보와 선택 묶음 상세 영역을 추가하거나 정리한다.
- 구현은 기존 GUI 모듈 안에서 진행하되, 애니메이션 전용 컨트롤러와 재생 상태 모델을 작은 내부 클래스로 분리한다.
- 애니메이션 상태 모델은 현재 시각, 재생 여부, 속도 배율, 선택 토큰, stale 여부, 축약 모드 여부를 소유한다.
- 애니메이션 컨트롤러는 시뮬레이션 결과와 현재 시각을 받아 캔버스에 표시할 토큰 상태를 계산한다.
- 캔버스 뷰는 블록/연결 레이어와 애니메이션 토큰 레이어를 분리해, 일반 다이어그램 편집과 애니메이션 렌더링이 서로 지우지 않도록 한다.
- 결과 패널은 애니메이션 컨트롤러가 제공하는 현재 시점 요약과 선택 묶음 정보를 표시한다.

## Testing Decisions

- 좋은 테스트는 사용자가 관찰할 수 있는 외부 동작을 검증한다. 내부 timer 호출 순서, canvas item ID, 임시 자료구조 모양에는 의존하지 않는다.
- 엔진 단위 테스트는 기존 상태를 유지한다. 이번 단계는 엔진 계약을 바꾸지 않으므로 기존 묶음/제품 라벨 테스트가 계속 통과해야 한다.
- 애니메이션 이벤트 계산은 GUI 렌더링과 분리된 작은 상태 계산 단위로 테스트한다.
- 애니메이션 상태 계산 테스트는 특정 시각에서 묶음이 도착 전, 대기 중, 처리 중, 완료 상태로 분류되는지 검증한다.
- 진행률 계산 테스트는 start/completion 시각 사이에서 0에서 1 사이 값이 안정적으로 계산되는지 검증한다.
- 속도 계산 테스트는 total_time이 다른 시나리오에서도 1x 기준 재생 시간이 약 30초 목표에 맞는지 검증한다.
- stale 상태 테스트는 블록 위치 이동과 구조/파라미터 변경을 구분하는 외부 동작을 검증한다.
- 제품명 기반 색상 배정은 같은 제품명이 같은 색을 얻고 다른 제품명은 안정적으로 구분되는지 검증할 수 있다.
- 축약 모드 테스트는 활성 토큰 개수가 기준을 넘을 때 집계 토큰이 생성되고 기준 이하에서는 개별 토큰이 유지되는지 검증한다.
- 기존 앱 포맷팅 테스트는 실제 연결 기반 흐름 표시가 유지되는지 계속 검증한다.
- tkinter canvas의 실제 픽셀 렌더링은 자동 테스트의 1차 대상이 아니다. B 단계 GUI 변경은 수동 스모크 체크리스트로 최종 확인한다.
- 수동 스모크 체크리스트에는 재생/일시정지/정지, 슬라이더 이동, 속도 선택, 토큰 선택, stale 상태, 축약 모드, 완료 상태, 병목 배지 확인을 추가한다.

## Further Notes

- 이번 PRD는 기존 단계 B의 넓은 후보 중 UI 개선과 중앙 다이어그램 애니메이션만 다룬다.
- 단계 A 재정의와 단계 C 제품 라벨 추적이 현재 활성 도메인 계약이다. 이전 EA push 기반 단계 A 문서는 역사적 참고로만 본다.
- 애니메이션은 계산 엔진의 대체물이 아니라 결과 해석 도구다. 시각 효과가 실제 공정 시간 의미를 새로 만들면 안 된다.
- 연결선 위 이동은 사용자의 시선 이동을 돕는 표현일 뿐이며, 실제 운반 시간은 HOIST 블록에서 계산된다.
- 사용자가 직접 배치한 다이어그램을 존중한다. 자동 배치는 별도 기능으로 남긴다.
- B 단계 구현 전 이 PRD를 구현 이슈로 쪼갤 때는 UI 구조 정리, playhead 상태 모델, 토큰 렌더링, 결과 패널 연동, 스모크 체크리스트 갱신 순서로 나누는 것이 적절하다.
